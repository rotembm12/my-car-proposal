The Problem

We audited all 91 screens of the current app. Here's what we found: friction at every stage of the customer journey, from first open to rental return.

03
My Car

Problem Area 1

Registration Dropout

The current registration spans 15+ screens across 3 separate parts. Each friction point is an exit opportunity.

3-Part Fragmented Onboarding
Part 1: Personal details form (name, address, ID). Part 2: Driving license info, contract viewing as a raw PDF, digital signature with no context ("you're signing the contract" never appears). Part 3: Document uploads (license photo, selfie, passport for non-Israelis) on separate screens. Each part feels like starting over.
Information Walls Before Action
Before users can even fill in details, they're shown 3 consecutive info screens: "Things to know about pricing" (tariff structure, Shabbat policy), "Important rules" (parking rules, fuel policy), and "Nice to meet you" (fill details). Each requires reading and tapping "Next". Users who just want to sign up are forced to read marketing content first.
Iframe Payment Experience
Credit card entry is handled via Tranzila iframe - a third-party payment page embedded inside the app. It uses a different visual style, has a different font, and doesn't match the app's UI. Users see a "verification charge of 100 NIS" warning. On mobile, iframe forms are notoriously hard to fill (auto-fill doesn't work, keyboard overlap issues). No Apple Pay or Google Pay alternative.
No Progress Indicator
Users have no idea how many steps remain. They see individual screens but have no sense of "I'm on step 3 of 5" or "almost done". This is a well-documented UX anti-pattern - without a progress bar, users overestimate remaining effort and abandon at higher rates.
Document Upload is Sequential & Blind
License photo and selfie upload happen on separate screens. There's no camera overlay guide showing users how to frame their license. The selfie screen shows a static example photo of a man holding an ID, but no live camera frame guide. If the photo is rejected later by admin, users must redo the entire upload - there's no inline validation or quality check.
Post-Registration Limbo
After completing all 15+ screens, users see "Thank you for registering! Your registration will be approved shortly" - with no estimated timeline, no status tracking, and no ability to use the app while waiting. They've invested significant time and are now locked out completely until manual admin approval. If details are wrong, they get a popup saying "some details already exist in the system" with no specifics about what to fix.

My Car

Problem Area 2

Booking & Rental Friction

Once registered, users face unnecessary complexity in finding, booking, and managing their rental.

8-Tap Booking Flow
To book a car, a user must: (1) tap "Book Vehicle" button, (2) select pickup location on map, (3) pick start date/time via date picker, (4) pick end date/time via separate date picker (no duration shortcut), (5) tap "Search", (6) wait for a loading animation, (7) view the found car's details and pricing, (8) confirm booking. Competitors like Zipcar and Share Now accomplish this in 2-3 taps via map marker > bottom sheet > book.
No Price Visibility Until the End
Users go through the entire booking flow before seeing what it will cost. There's no price estimator, no "starting from X/hour" on the map, and no cost preview during car selection. The pricing table exists as a separate menu item with a complex matrix of per-category hourly and per-km rates, seasonal surcharges, and add-ons - but it's disconnected from the booking flow. Users must mentally calculate their expected cost.
Hidden Unlock Code + Non-Standard Gesture
During an active rental, the car's unlock code is hidden behind a "slide right to reveal" gesture. This is not a standard mobile interaction pattern - users standing at the car, possibly in a hurry or in bad weather, must discover this gesture on their own. There is no tooltip, no animation hint, and no fallback. If a user taps instead of slides, nothing happens. The code should be the most prominent element on the active rental screen.
No Real-Time Cost During Rental
While renting, users see a countdown timer but have zero visibility into how much they're spending. The pricing model is complex (per-hour base + per-km charge), yet the app shows no running total. Users only discover the actual cost after returning the car. This creates anxiety and distrust, especially for first-time users. The order summary screen at the end is dense with line items but comes too late to influence behavior.
Round-Trip Only, No One-Way
Users must return the car to the same station they picked it up from. This is a significant limitation for point-to-point use cases (e.g., driving from home to the airport). Competitors increasingly support one-way drops, and the lack of this feature eliminates an entire category of potential bookings.
No Damage Documentation
There is no pre-rental or post-rental damage reporting flow. Users have no way to document existing damage before pickup, which means they can be held responsible for damage they didn't cause. This creates disputes, requires manual investigation by support, and erodes user trust. Industry standard for shared mobility is mandatory photo inspection before and after each rental.

Problem Area 3

Platform & Design Debt

Systemic issues that affect every interaction - from visual identity to navigation to cross-platform consistency.

Outdated Visual Identity
The app uses a flat mint/emerald green color scheme with no depth, shadows, or visual hierarchy. Buttons are solid-color rounded rectangles with no states (pressed, disabled, loading). Modal dialogs resemble browser alerts (plain white box, centered text, single button) rather than native components. Car illustrations are generic clip-art style. There is no dark mode support despite ~80% of mobile users preferring dark mode at night. The overall look communicates "built in 2018" rather than "premium shared mobility service".
Hamburger Menu Hides Core Features
Primary actions are buried in a slide-out side menu: profile, history, future bookings, pricing, support, and terms. There are no bottom tabs. Users must tap the hamburger icon, scan the menu, and tap again to reach anything beyond the map. UX research consistently shows that hamburger menus reduce feature discoverability by 50%+ compared to visible tab bars. Key features like "My Bookings" should be one tap away, not two.
No Biometric Authentication
Every login requires phone number + OTP via SMS. There is no Face ID, Touch ID, or fingerprint option. For returning users who open the app daily, this means waiting for an SMS every time. The OTP has a 30-second retry cooldown, adding further delay. Most modern apps offer biometric login after initial setup, reducing re-authentication to under 1 second.
Web Views Break the Native Experience
At least 3 critical screens are not native: (1) The pricing table opens as an embedded web page with a completely different layout and typography. (2) The terms of use redirect to the company website in an in-app browser. (3) The payment form is a Tranzila iframe. These transitions are jarring - users lose the app's navigation, back button behavior changes, and the visual language shifts. Each web view is a moment where users feel they've "left" the app.
Web App Cannot Complete Core Flows
The web application is limited - it handles registration but cannot perform the full booking flow, view active rentals, or manage all profile features. Users who prefer desktop (common for pre-planning rentals) or who can't install the app are underserved. This also means the web can't serve as a fallback when the mobile app has issues. A modern shared mobility service should offer full functionality on both platforms.
Profile Management is Scattered & Restrictive
Personal info, license details, credit card management, and contact info are all separate menu items leading to separate screens. Updating phone number or email requires contacting support (users see "to update phone or email, contact customer service") - there's no self-service option. The profile screen shows data in read-only mode with small pencil icons per section, each opening a different edit flow. Users who need to update an expired license must navigate through 3 screens to reach the upload form.
04c

The Solution

A complete client-side rewrite with consolidated flows, modern design system, and full platform parity. Same backend, entirely new experience.

Roadmap

Phase 1: MVP ~20 weeks

The complete core rental experience. Users can register, find, book, rent, report damage, and pay. All estimates include writing tests, QA cycles, and cross-platform verification (iOS + Android + Web).

Module What's Included & Why This Estimate Dev Weeks
Project Setup & Design System Week 1: Monorepo scaffold (Turborepo/Nx), Expo project init, Next.js project init, shared package structure (@mycar/ui, @mycar/api, @mycar/shared), CI/CD pipeline, linting/formatting config. Week 2-3: Core component library from Figma tokens - buttons (5 variants), text inputs (with validation states), cards, bottom sheets, modals, navigation shell (bottom tabs), RTL layout engine, typography scale, color system with theme context. This is the foundation everything else builds on - investing 3 weeks here pays off in every subsequent module. 3
Auth Week 1: Phone input with country code, OTP request/verify API integration, SMS auto-fill, 6-digit code input component, resend timer, error handling (invalid code, expired, rate limit). Week 2: Biometric enrollment flow (Face ID / Touch ID via Expo LocalAuthentication), secure token storage (Expo SecureStore on mobile, httpOnly cookies on web), session refresh logic, logout, unapproved-user gating (read-only map access, "Book" button shows approval status). 2 weeks because biometric + secure storage + session management across 3 platforms (iOS/Android/Web) requires careful implementation. 2
Registration Week 1: Multi-step form architecture with progress stepper, local state persistence (AsyncStorage), step 1 (personal details form: name, DOB, gender with validation), step 2 (address with Google Maps Places autocomplete, city/street/number auto-parse). Week 2: Step 3 (guided camera for license photo with overlay frame, selfie capture with face detection hint, photo preview & retake), passport flow variant for non-Israeli IDs. Week 3: Step 4 (rules carousel - swipeable cards with key policies), step 5 (contract viewer with scroll-to-bottom requirement + signature pad component), form submission, success/error states, API integration for all steps. 3 weeks because the camera overlay, signature pad, and Google Maps integration each involve native module configuration that takes longer than typical UI work. 3
Map & Discovery Week 1: Map integration (provider-agnostic wrapper), user location permissions, car marker rendering with clustering (100+ markers must perform smoothly), color-coded markers by category (economy=blue, standard=green, premium=gold, electric=teal), marker tap interaction. Week 2: Car detail bottom sheet (photo, model, category, station name, walking distance, per-hour rate), snap points (peek/half/full), list view alternative with sort controls. Week 3: Filter chip bar (car type multi-select, price range, distance radius), location search bar with Places autocomplete, polling mechanism for real-time availability updates (configurable interval), empty states, loading skeletons. 3 weeks because map performance tuning with many markers, the bottom sheet interaction, and filter logic with live polling require careful coordination. 3
Booking Week 1: "Book Now" flow (immediate booking from bottom sheet), scheduled booking (date/time pickers for start + end, min/max duration validation), one-way rental (drop-off station selector with map + list view of valid return locations). Week 2: Price estimator (calculate estimated cost based on duration + average km, show breakdown before confirmation), confirmation screen (all booking details + booking number), push notification scheduling (booking confirmed, reminder before start). Week 3: "My Bookings" tab (upcoming list with status badges), booking detail view, extend end time, cancel booking (with fee calculation and confirmation), toggle self-participation insurance, API integration for all CRUD operations. 3 weeks due to the number of distinct flows (immediate/scheduled/one-way) plus the management screens (list/detail/extend/cancel). 3
Active Rental Week 1: Rental dashboard layout (prominent unlock code with copy button, countdown timer with color transitions green>orange>red, car photo + model + station info), "Navigate to car" deep link (detect Apple Maps / Google Maps / Waze, pass coordinates). Week 2: Extend rental (in-line time picker, availability check, price update), end rental flow (confirmation dialog with trip summary, "car returned" success state), state management (dashboard replaces Map tab when rental is active, persistent banner when navigating away). 2 weeks because it's a single screen with well-defined interactions. The complexity is in state management (active rental state persists across app restarts and tab switches), not in the number of screens. 2
Damage Reporting Week 1: Guided photo capture flow (5 sequential steps: front, rear, left side, right side, interior), camera component with overlay frame showing expected composition for each angle, photo preview with accept/retake, progress indicator (3/5 done). Week 2: Optional text notes per photo, before-pickup enforcement (must complete before code is shown), after-return enforcement (must complete before rental is closed), background photo upload with progress bar, retry on failure, association with rental ID in API. 2 weeks because the camera overlay component needs to work on both iOS and Android (different camera APIs), and the enforcement logic (blocking code reveal / rental close until photos are submitted) needs careful state management. 2
Profile & Payments Week 1: Unified profile screen (bottom tab), personal info section (view + inline edit), license section (view details + upload new photo on renewal), address update with autocomplete, self-service phone/email change (with OTP verification). Week 2: Payment provider abstraction layer (PaymentProvider interface with tokenizeCard, getPaymentMethods, deletePaymentMethod), first provider implementation, Apple Pay + Google Pay integration (Expo or Stripe SDK), native card entry form, payment methods list (add/remove/set default), credits display. 2 weeks because the payment abstraction layer needs to be designed for multi-provider support from the start. The actual card UI is straightforward, but the provider-switching architecture requires upfront design. 2

Web parity is built in parallel via shared monorepo components. ~2 weeks are embedded in the estimates above for web-specific work (responsive breakpoints, SSR considerations, browser-specific testing). Each module's estimate includes unit/integration tests for that module's core flows.
