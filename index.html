The Problem

We audited all 91 screens of the current app. Here's what we found: friction at every stage of the customer journey, from first open to rental return.

03
x My Car

Problem Area 1

Registration Dropout

The current registration spans 15+ screens across 3 separate parts. Each friction point is an exit opportunity.

3-Part Fragmented Onboarding
Registration split across 3 disconnected parts - each feels like starting over.
Part 1: Personal details form. Part 2: License info, raw PDF contract, signature with no context. Part 3: Document uploads (license photo, selfie, passport) on separate screens.
Information Walls Before Action
3 consecutive info screens force users to read before they can sign up.
"Things to know about pricing", "Important rules", "Nice to meet you" - each requires reading and tapping "Next". Users who just want to register are blocked by marketing content.
Iframe Payment Experience
Credit card entry via third-party iframe - different UI, no Apple/Google Pay.
Tranzila iframe uses different visual style and font. Shows "verification charge of 100 NIS" warning. On mobile, auto-fill doesn't work and keyboard overlaps the form.
No Progress Indicator
Users have no idea how many steps remain - leads to higher abandonment.
No "step 3 of 5" or progress bar. Without visible progress, users overestimate remaining effort and abandon at higher rates.
Document Upload is Sequential & Blind
No camera guides, no inline validation - rejected photos require full redo.
License photo and selfie on separate screens. No camera overlay guide for framing. If rejected by admin later, users must redo the entire upload with no specifics about what was wrong.
Post-Registration Limbo
After 15+ screens, users are locked out with no timeline or status tracking.
"Your registration will be approved shortly" - no estimated time, no status page, no ability to browse the app while waiting. Errors show "details already exist" with no specifics.

Problem Area 2

Booking & Rental Friction

Once registered, users face unnecessary complexity in finding, booking, and managing their rental.

8-Tap Booking Flow
Booking requires 8 taps across disconnected screens. Competitors do it in 2-3.
(1) Tap "Book Vehicle", (2) select location, (3) pick start date, (4) pick end date, (5) tap "Search", (6) wait for loading, (7) view car details, (8) confirm. Zipcar/Share Now: map marker > bottom sheet > book.
No Price Visibility Until the End
Users complete the entire booking flow before seeing what it costs.
No price estimator, no "from X/hour" on map markers. Pricing table exists as a separate menu item with complex per-category rates - disconnected from booking. Users must mentally calculate.
Hidden Unlock Code
Car code hidden behind "slide right" - a non-standard gesture users must discover.
No tooltip, no animation hint. If users tap instead of slide, nothing happens. Standing at the car in a hurry, this creates frustration. The code should be the most prominent element on screen.
No Real-Time Cost During Rental
Zero cost visibility while renting - users discover the total only after returning.
Complex pricing (per-hour + per-km), yet no running total shown. Creates anxiety and distrust, especially for first-time users. Summary comes too late to influence behavior.
Round-Trip Only, No One-Way
Must return to same station. Eliminates point-to-point use cases entirely.
No support for picking up at A and dropping at B (e.g., home to airport). Competitors increasingly support one-way, and lacking it removes an entire booking category.
No Damage Documentation
No pre/post-rental photo inspection. Users can be blamed for pre-existing damage.
No way to document damage before pickup. Creates disputes, manual support investigations, and erodes trust. Industry standard is mandatory photo inspection before and after each rental.

Problem Area 3

Platform & Design Debt

Systemic issues that affect every interaction - from visual identity to navigation to cross-platform consistency.

Outdated Visual Identity
Flat green scheme, no depth, no dark mode. Looks "built in 2018".
Buttons with no states (pressed/disabled/loading). Modal dialogs resemble browser alerts. Generic clip-art car illustrations. No dark mode despite ~80% of users preferring it at night.
Hamburger Menu Hides Core Features
Primary actions buried in side menu. No bottom tabs - everything is 2+ taps away.
Profile, history, bookings, pricing, support all hidden in hamburger. Research shows hamburger menus reduce discoverability by 50%+ vs visible tab bars.
No Biometric Authentication
Every login requires SMS OTP. No Face ID or fingerprint option.
Returning users wait for SMS every time. 30-second retry cooldown adds delay. Modern apps reduce re-auth to under 1 second via biometrics.
Web Views Break Native Experience
3 critical screens open as embedded web pages - jarring transitions, broken navigation.
Pricing table, terms of use, and payment form all use different layouts and typography. Users lose app navigation and feel they've "left" the app.
Web App Cannot Complete Core Flows
Web handles registration only - no booking, no rentals, no full profile management.
Desktop users (common for pre-planning) and users who can't install the app are underserved. Web can't serve as fallback when mobile has issues.
Profile Management is Scattered
Personal info, license, cards, contact - all separate screens. Phone/email changes require support.
No self-service for contact updates ("contact customer service"). Read-only profile with small pencil icons opening different edit flows. License renewal requires navigating 3 screens.

Delivery Model

Design Leads, Dev Follows

UI/UX and development run as two parallel tracks. Design works ~2-3 weeks ahead of development, delivering Figma handoffs that feed directly into implementation sprints.

🎨
UI/UX Design Track
Starts from scratch - no existing Figma, flow docs, or design assets.
  • Discovery & audit come first - validate assumptions before designing
  • Design system (tokens, components) built once, used everywhere
  • Each flow delivered as annotated Figma with behavior notes & spacing
  • 2 revision rounds included per deliverable, 20% buffer per phase
  • Weekly sync with project owner + stakeholder for feedback
💻
Development Track
1 frontend developer, full-time. Builds against finalized Figma handoffs. QA validates each module.
  • Monorepo setup & design system implementation starts in parallel with UX discovery
  • Each dev sprint starts after corresponding design handoff is approved
  • Shared component library (React Native + Web) built from Figma tokens
  • Tests written per module, QA cycle after each major flow
  • Web parity built in parallel via shared components

Design leads by 2-3 weeks - dev is never blocked, design gets feedback before next handoff.
More efficient than waterfall (all design then all dev) or fully parallel (constant context switching). Each handoff is approved before dev starts that module.

Testing Strategy

Test external behavior, not implementation details. Prefer integration tests over unit tests for UI flows.

  • Auth flow: OTP, biometrics, session, approval gating
  • Registration: full 5-step E2E, validation, persistence
  • Booking: immediate, scheduled, one-way, cancel, extend
  • Damage reporting: all 5 angles, skip prevention, upload retry
  • Payment: provider abstraction, card entry, Apple/Google Pay

Rollout

Migration Strategy

Continuity

  • Published as update to existing app listings (preserves ratings, downloads, store presence)
  • Existing users log in with same phone number and see full history
  • Feature flag rollout - gradually enable for user segments
  • Rollback plan: keep old app version deployable during transition

Dependencies & Assumptions

  • UI/UX Brand Guidelines: Design system depends on guidelines currently in development. Can start with placeholder tokens.
  • API Documentation: Comprehensive docs for all existing backend endpoints required before dev starts.
  • Existing APIs: All backend APIs assumed to exist (auth, booking, fleet, payments, push). Missing APIs are a separate backend scope.
  • App Store Accounts: Existing Apple Developer and Google Play Console accounts.
  • No Backend Work: This proposal covers client-side only. Backend modifications (new endpoints, schema changes) are out of scope.
